# DevSpaces Implementation Comparison

**ansible-devspaces-summit vs ansible/ansible-dev-tools**

Date: 2026-03-31

---

## Overview

This report compares two DevSpaces implementations for Ansible development environments:

- [**ansible-devspaces-summit**](https://github.com/leogallego/ansible-devspaces-summit) — Custom all-in-one repo with image, devfile, and VS Code workspace config. An Ansible workspace for DevSpaces with nested containers support.
  - [devfile.yaml](https://github.com/leogallego/ansible-devspaces-summit/blob/main/devfile.yaml)
  - [Containerfile](https://github.com/leogallego/ansible-devspaces-summit/blob/main/workspace-image/Containerfile)
  - [devspaces.code-workspace](https://github.com/leogallego/ansible-devspaces-summit/blob/main/devspaces.code-workspace)
- [**ansible/ansible-dev-tools**](https://github.com/ansible/ansible-dev-tools) — Upstream repo maintained by the Ansible DevTools team. The `devspaces/` directory builds the `ghcr.io/ansible/ansible-devspaces` image, and `devfile.yaml` provides the reference workspace definition.
  - [devfile.yaml](https://github.com/ansible/ansible-dev-tools/blob/main/devfile.yaml)
  - [devspaces/Containerfile](https://github.com/ansible/ansible-dev-tools/blob/main/devspaces/Containerfile)
  - [devspaces/context/setup.sh](https://github.com/ansible/ansible-dev-tools/blob/main/devspaces/context/setup.sh)
  - [devspaces/context/entrypoint.sh](https://github.com/ansible/ansible-dev-tools/blob/main/devspaces/context/entrypoint.sh)

As of [commit 06600bd](https://github.com/ansible/ansible-dev-tools/commit/06600bd44c40e87254cc65d24ef647d549072876), both repos now use the same core approach: **rootless podman via user namespaces** for container-in-container support. Neither uses kubedock.

| | **ansible-devspaces-summit** | **ansible-dev-tools** |
|---|---|---|
| Schema version | 2.2.0 | 2.2.2 |
| Workspace name | `ansible-devspaces-example` | `ansible-demo` |
| Components | 2 (`dev-tools` + `prep-workspace` init container) | 1 (`tooling-container`) |
| Container image | `quay.io/leogallego/ansible-devspaces-summit:latest` | `ghcr.io/ansible/ansible-devspaces:latest` |
| CPU request / limit | 250m / 4000m | 250m / 2000m |
| Memory request / limit | 128Mi / 4Gi | 256M / 6Gi |
| Storage type | `per-workspace` (explicit) | Default (not specified) |
| Pod/container overrides | Explicit — `hostUsers: false`, `procMount: Unmasked`, CRI-O device annotations (`/dev/fuse`, `/dev/net/tun`) | None — relies on Dev Spaces operator (3.25+ / OCP 4.20+) to inject automatically |
| Init container | Copies `oc` and `kubectl` from internal OpenShift image registry | None — CLI tools baked into image via `setup-image.sh` |
| `VSCODE_DEFAULT_WORKSPACE` | Set (loads `.code-workspace`) | Not set |
| `ANSIBLE_COLLECTIONS_PATH` | Not set | Set to 3 paths including project collections |
| `PATH` override | Prepends `/projects/bin` for oc/kubectl | Not set |
| Devfile commands | None | 6 Molecule commands (create, list, converge, verify, destroy, test) + ansible-navigator launcher |
| preStop event | None | `molecule-destroy` (auto-cleanup) |

### Notable differences

- **Pod overrides:** Our devfile includes explicit `pod-overrides` and `container-overrides` attributes, making it compatible with **Dev Spaces 3.20+ / OCP 4.17+**. Upstream removed theirs and now requires **Dev Spaces 3.25+ / OCP 4.20+** where the operator handles injection automatically.
- **Devfile commands:** Upstream provides a guided Molecule testing workflow with numbered steps and an ansible-navigator launcher. Our devfile has no commands defined.
- **Init container:** We use a separate init container to copy OpenShift CLI binaries. Upstream installs `oc` directly into the image, eliminating the extra component.

### UID Mapping: Static vs Dynamic

This is the most significant technical difference between the two images.

**Our repo (static):**
- UID 1000 is hardcoded at build time
- `subuid`/`subgid` entries are fixed: `user:1001:64535`
- Overlay storage config is written to `~/.config/containers/storage.conf` at build time
- If OpenShift assigns a UID other than 1000, rootless podman will fail

**Upstream (dynamic):**
- Entrypoint runs at container start and detects the actual UID assigned by the platform
- Reads `/proc/self/uid_map` to determine namespace size
- Calculates available subordinate IDs dynamically
- Writes `subuid`/`subgid` on every start with the correct values
- Includes error handling for unwritable files and insufficient subordinate IDs
- Works regardless of what UID OpenShift assigns

## Companion Repo: [redhat-developer-demos/ansible-devspaces-demo](https://github.com/redhat-developer-demos/ansible-devspaces-demo)

The `ansible-devspaces-demo` repo is the intended **consumer** of the upstream image. It does not build an image — it uses `ghcr.io/ansible/ansible-devspaces:latest` directly and adds:

- Sample Ansible content (`sample_namespace/sample_collection` with a `backup_file` role)
- Molecule test configurations
- `.ansible-lint` and `.yamllint` configs
- `.pre-commit-config.yaml` (pre-commit hooks, ansible-lint, gitleaks)
- `.vscode/` settings and extensions
- `.code-workspace` with Lightspeed enabled
- `KUBEDOCK_ENABLED=true` (note: this may be outdated now that upstream moved to nested podman)

This represents the pattern upstream expects: **image repo provides the tooling, project repo provides the content and editor config**.

## Summary

### What our repo has that upstream doesn't

| Capability | Benefit |
|---|---|
| Explicit pod/container overrides in devfile | Works on older OCP 4.17+ / Dev Spaces 3.20+ |
| OpenCode AI tool in image | AI-assisted development in the workspace |
| VS Code workspace config | Zero-config editor experience for Ansible |
| Fully pinned dependency lockfile | Reproducible builds across environments |
| All-in-one repo (image + devfile + workspace) | Single source of truth, simpler to manage |

### What upstream has that we don't

| Capability | Benefit |
|---|---|
| Dynamic UID mapping in entrypoint | Works with any UID OpenShift assigns; prevents rootless podman failures |
| Molecule devfile commands | Guided testing workflow for demos and onboarding |
| preStop molecule-destroy | Automatic resource cleanup on workspace stop |
| Pre-installed Ansible collections | Ready to use without `ansible-galaxy install` |
| Ansible-themed shell prompt | Better developer experience with git branch visibility |
| Build caching (`--mount=type=cache`) | Faster image rebuilds |
| Python 3.12 | Current Python version |
| `oc` baked into image | No init container needed, simpler devfile |
| Full OCI labels | Better image metadata and discoverability |

## Recommended Actions

### High priority

1. **Adopt the dynamic UID entrypoint from upstream.** The static UID 1000 mapping is the most significant risk — if OpenShift assigns a different UID, rootless podman silently breaks. Upstream's entrypoint handles this gracefully.

2. **Add Molecule and navigator devfile commands.** Low effort, high value for demos. Can be copied almost directly from upstream's `devfile.yaml`.

3. **Add `preStop: molecule-destroy`.** Prevents dangling test pods when workspaces are stopped.

### Medium priority

4. **Bump Python to 3.12** and devfile schema to 2.2.2.

5. **Pre-install common Ansible collections** (`ansible.posix`, `ansible.utils`, `ansible.netcommon`, `ansible.scm`).

6. **Add build caching** to the Containerfile (`--mount=type=cache` for dnf and pip layers).

7. **Consider baking `oc`/`kubectl` into the image** instead of using an init container, simplifying the devfile to a single component.

### Low priority

8. Add a custom shell prompt with git branch indicator.

9. Add full OCI labels to the Containerfile.

10. Add `.ansible-lint` and `.yamllint` configs to the repo.
